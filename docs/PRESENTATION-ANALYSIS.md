# วิเคราะห์: สิ่งที่ปรากฏบนเว็บ vs สิทธิ์ถูกกรรมการถาม

เอกสารนี้สรุปว่าส่วนไหนบนเว็บอาจถูกกรรมการซักถาม ส่วนไหนไม่จำเป็นสำหรับโปรเจกต์จบ และแนะนำว่าตัดออกหรือลดความเสี่ยงได้อย่างไร

---

## 1. หน้าที่แสดง "ปิดใช้งานชั่วคราว" — **แนะนำ: ซ่อนหรือตัดออก**

| หน้า | URL | สถานะปัจจุบัน | ความเสี่ยง |
|------|-----|----------------|------------|
| เกี่ยวกับเรา | `/about` | แสดงข้อความ "หน้านี้ปิดใช้งานชั่วคราว" | กรรมการอาจถามว่าทำไมปิด / ทำไม่เสร็จ |
| เอกสาร API | `/api-docs` | แสดงข้อความ "หน้านี้ปิดใช้งานชั่วคราว" | เหมือนกัน |

**คำแนะนำ**

- **ตัวเลือก A (แนะนำ):** ไม่ให้ผู้ใช้เห็นหน้านี้เลย  
  - ทำ **redirect** จาก `/about` และ `/api-docs` ไปยัง `/` หรือ `/dashboard`  
  - จะไม่มีคำว่า "ปิดใช้งานชั่วคราว" ให้กรรมการกดเข้าไปเห็น  
- **ตัวเลือก B:** เก็บ route ไว้แต่ **ไม่ใส่ลิงก์** ไปที่หน้านี้จากที่ใดในเว็บ (ตอนนี้ก็ไม่มีลิงก์ใน footer แล้ว)  
  - ยังมีความเสี่ยงอยู่ถ้ากรรมการพิมพ์ URL เอง

**สรุป:** ส่วนที่ "ไม่จำเป็น" และตัดออกได้ชัดเจนคือ **การให้เข้าถึงหน้า about / api-docs ที่ว่างหรือปิดใช้งาน** — แนะนำให้ redirect ออกไปเลย

---

## 2. Footer: ติดต่อสอบถาม (ที่อยู่ + อีเมล)

| รายการ | ที่แสดง | ความเสี่ยง |
|--------|---------|------------|
| ที่อยู่ | 80 ถ.นครสวรรค์ ต.ตลาด อ.เมือง จ.มหาสารคาม 44000 | อาจถูกถามว่าเป็นที่อยู่ทางการของมหาวิทยาลัยหรือไม่ |
| อีเมล | contact@rmu.ac.th | อาจถูกถามว่าเป็นอีเมลหน่วยงานจริงหรือเป็นตัวอย่าง |

**คำแนะนำ**

- ถ้าเป็น **ข้อมูลตัวอย่าง** เพื่อให้เว็บดูสมบูรณ์: ควรใส่คำว่า "ตัวอย่าง" หรือ "ติดต่อผ่านช่องทางที่มหาวิทยาลัยกำหนด"  
- ถ้าเป็น **ข้อมูลจริงที่ได้รับอนุญาต**: เก็บได้ แค่เตรียมอธิบายว่าได้มาจากที่ไหน  
- ถ้าไม่ต้องการถูกถามเรื่องนี้: **ตัดบล็อก "ติดต่อสอบถาม" ออกจาก footer** ได้ (เหลือแค่ FAQ, ข้อกำหนด, นโยบาย, แนวทาง)

**สรุป:** ส่วนนี้ **ไม่จำเป็น** สำหรับการสาธิตโปรเจกต์จบ ถ้าต้องการลดคำถามแนะนำตัดหรือทำให้ชัดว่าเป็นตัวอย่าง

---

## 3. ฟีเจอร์ LINE (แจ้งเตือนผ่าน LINE)

| ที่แสดง | ความเสี่ยง |
|---------|------------|
| หน้าแรก: การ์ด "แจ้งเตือนผ่าน LINE" | กรรมการอาจถามว่า implement จริงไหม / ใช้ API อะไร |
| หน้าโปรไฟล์: เชื่อมบัญชี LINE | เหมือนกัน |
| Admin: LINE Test | กรรมการที่เป็น admin อาจเห็น |

**คำแนะนำ**

- ถ้า **ทำจริง** (มี LINE Messaging API / Webhook): เก็บไว้ได้ แค่เตรียมอธิบายสั้นๆ ว่าใช้ LINE API ทำอะไร (แจ้งเตือนเมื่อมีคำขอ/สถานะเปลี่ยน ฯลฯ)  
- ถ้า **ยังไม่ลงมือหรือเป็น mock**: ควรเตรียมคำตอบว่าส่วนนี้เป็นฟีเจอร์ที่ออกแบบไว้ หรือกำลังพัฒนา  
- **ไม่จำเป็นต้องตัดออก** ถ้าคุณพร้อมอธิบายได้

---

## 4. Help Bot (ชาร์คกี้ / AI แชท)

| ที่แสดง | ความเสี่ยง |
|---------|------------|
| ปุ่มแชทช่วยเหลือ (พี่ชาร์คกี้) | กรรมการอาจถามว่าใช้ AI ตัวไหน (Gemini), ค่าใช้จ่าย, นโยบายการใช้ข้อมูล |

**คำแนะนำ**

- ถ้าต้องการ **ลดประเด็นที่กรรมการถาม**:  
  - **ซ่อนปุ่ม Help Bot** เฉพาะช่วงนำเสนอ (เช่น ใช้ env หรือ flag `NEXT_PUBLIC_SHOW_HELP_BOT=false` แล้วไม่แสดงปุ่ม)  
- ถ้าพร้อมอธิบาย: เก็บไว้ได้ แค่เตรียมคำตอบว่าใช้ Google Gemini ผ่าน API ทำอะไร (ตอบคำถามเกี่ยวกับการใช้งานใน RMU)

**สรุป:** ส่วนนี้ **ไม่จำเป็น** สำหรับการพิสูจน์ว่าแอป "แลกเปลี่ยนของ" ทำงานครบ — ตัดหรือซ่อนได้ถ้าอยากลดคำถาม

---

## 5. Analytics / Vercel / Sentry

| รายการ | ที่แสดงบนเว็บ | ความเสี่ยง |
|--------|----------------|------------|
| Vercel Analytics | ไม่มีข้อความบนหน้า (มีแค่ script) | น้อยมาก อาจถามว่าใช้บริการอะไรบ้าง |
| Sentry | ไม่มีข้อความบนหน้า | เหมือนกัน |

**คำแนะนำ**

- **ไม่จำเป็นต้องตัดออก** สำหรับวันนำเสนอ  
- ถ้าถูกถาม: บอกได้ว่าใช้เพื่อวิเคราะห์การใช้งาน (Vercel) และรายงาน error (Sentry) เป็นเครื่องมือพัฒนา

---

## 6. ข้อกำหนด / นโยบาย (Terms, Privacy, Guidelines)

| หน้า | ความจำเป็นสำหรับโปรเจกต์จบ |
|------|-----------------------------|
| ข้อกำหนดการใช้งาน (/terms) | **จำเป็น** — ใช้ร่วมกับ consent ก่อนใช้งาน |
| นโยบายความเป็นส่วนตัว (/privacy) | **จำเป็น** — มาตรฐานและจริยธรรม (PAPA ฯลฯ) |
| แนวทางชุมชน (/guidelines) | **ควรมี** — แสดงว่าระบบมีกติกา |

**คำแนะนำ:** **ไม่ควรตัดออก** และควรเตรียมอธิบายสั้นๆ ว่าใช้กับ consent อย่างไร

---

## 7. Admin Panel

| ที่แสดง | ความเสี่ยง |
|---------|------------|
| ปุ่ม "Admin" ใน navbar | แสดงเฉพาะเมื่อ `isAdmin` | กรรมการที่เป็น admin อาจถามว่ามีฟีเจอร์อะไรบ้าง |

**คำแนะนำ**

- **เก็บไว้** ถ้าจะสาธิตหน้าผู้ดูแล (รายงาน, ผู้ใช้, สถิติ ฯลฯ)  
- เตรียมอธิบายว่า Admin ใช้ทำอะไร (จัดการคำร้อง, รายงาน, สถิติ)  
- **ไม่จำเป็นต้องตัดออก**

---

## 8. อื่นๆ ที่อาจถูกถาม (แต่ไม่ต้องตัดออก)

- **อีเมล @rmu.ac.th** — กรรมการอาจถามว่าเช็ค domain จริงไหม → เตรียมตอบว่าใช้ Firebase Auth จำกัดอีเมล  
- **สถิติบนหน้าแรก** (จำนวนสิ่งของ, ผู้ใช้, แลกเปลี่ยนสำเร็จ) — อาจถามว่าข้อมูลจริงไหม → ตอบว่าเรียกจาก API / ฐานข้อมูลจริง  
- **รายงาน (Report)** — อาจถาม flow การรายงาน → อธิบายได้ว่า user รายงานแล้ว admin ดูที่หน้าจัดการรายงาน  

---

## สรุป: อะไรตัดออกได้บ้าง

| ลำดับ | รายการ | แนะนำ |
|-------|--------|--------|
| 1 | หน้า **About** และ **API Docs** ที่แสดง "ปิดใช้งานชั่วคราว" | **Redirect ไปหน้าหลัก/แดชบอร์ด** เพื่อไม่ให้มีหน้าร้างให้กรรมการกดเข้า |
| 2 | **Footer: ติดต่อสอบถาม** (ที่อยู่ + อีเมล) | **ตัดออกได้** หรือใส่คำว่า "ตัวอย่าง" ถ้าต้องการลดคำถาม |
| 3 | **Help Bot (ชาร์คกี้)** | **ซ่อนได้** ด้วย env/flag ถ้าไม่ต้องการให้กรรมการโฟกัสที่ AI |
| 4 | Terms / Privacy / Guidelines | **ไม่ตัด** — จำเป็นสำหรับ consent และจริยธรรม |
| 5 | LINE, Admin, Analytics | **ไม่ต้องตัด** — เตรียมคำตอบอธิบายสั้นๆ ก็เพียงพอ |

---

## สรุปหนึ่งบรรทัด

- **ตัด/ซ่อนได้ชัดเจน:** (1) หน้า about + api-docs ที่ปิดใช้งาน → ทำ redirect  
- **ตัดได้ถ้าต้องการลดคำถาม:** (2) บล็อกติดต่อใน footer, (3) ปุ่ม Help Bot  
- **ไม่ควรตัด:** Terms, Privacy, Guidelines, ฟีเจอร์หลัก (โพสต์, แลกเปลี่ยน, แชท, รายงาน)

---

## สรุป: สิ่งที่ทำแล้วตามคำแนะนำ

| รายการ | สถานะ |
|--------|--------|
| **Redirect /about และ /api-docs** | ✅ ทำแล้ว — ทั้งสอง redirect ไป `/` จะไม่มีข้อความ "ปิดใช้งานชั่วคราว" ให้กรรมการเห็น |
| **Footer: ตัดบล็อกติดต่อสอบถาม** | ✅ ทำแล้ว — ลบคอลัมน์ที่อยู่ + อีเมลออก เหลือ FAQ + นโยบายและความปลอดภัย |
| **Help Bot** | ไม่ได้ซ่อนด้วย env — ถ้าในแอปมีปุ่มชาร์คกี้และอยากซ่อนช่วงนำเสนอ ค่อยเพิ่ม `NEXT_PUBLIC_SHOW_HELP_BOT=false` แล้วเช็คใน layout ได้ |
